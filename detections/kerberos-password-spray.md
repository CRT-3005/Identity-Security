# 🔐 Kerberos Password Spray

This detection demonstrates how Kerberos password spray activity against Active Directory can be identified in Splunk using Windows Security Event Logs.

After validating log ingestion from the Domain Controller (`ADDC01`) and configuring the required audit policies, a Kerberos password spray was simulated from the Kali attacker machine using **Kerbrute**. Splunk was then used to identify and analyse the authentication events generated by that activity.

---

## 🎯 Objective

The objective of this detection is to simulate and identify a Kerberos-based identity attack by generating a Kerberos AS-REQ authentication attempt with an invalid password using **Kerbrute** from the attacker system (`192.168.10.250`).

This demonstrates how identity-centric authentication activity appears in Splunk and how analysts can detect Kerberos-based credential guessing attempts.

---

## ⚔️ Attack Execution (Kali Linux)

A controlled password spray was conducted using `kerbrute` with a small user list and a delay to reduce the chance of account lockout.

```bash
echo "JNeutron" > /tmp/users.txt

kerbrute passwordspray \
  --dc 192.168.10.7 \
  -d ADProject.local \
  --delay 1000 \
  --safe \
  -v \
  /tmp/users.txt Welcome123
```

## 🔍 Detection in Splunk

To maintain visibility even when EventCode extraction varies, a regex-assisted SPL query was used to extract Kerberos authentication events directly from the raw Security XML logs.

This approach is reliable when Windows fields are not consistently normalized.

### Splunk Search (Regex-Assisted)

```spl
index=identity host=ADDC01 source="WinEventLog:Security" earliest=-10m
| rex field=_raw "<EventID>(?<EventID>\d+)</EventID>"
| rex field=_raw "(?i)<Data Name=\"TargetUser(?:Name|Name)\">(?<TargetUserName>[^<]+)"
| rex field=_raw "(?i)<Data Name=\"IpAddress\">(?<IpAddress>[^<]*)"
| where _indextime>=relative_time(now(), "-5m")
| search EventID=4768 OR EventID=4771
| table _time _indextime EventID TargetUserName IpAddress host
| sort -_indextime
```

This confirms that the Kerberos password spray attempt generated authentication activity on the Domain Controller and that the event was successfully ingested and analysed in Splunk.

## ⚙️ Audit Policy Configuration (Domain Controller)

To capture Kerberos-related identity logs reliably, the following advanced audit subcategories were enabled:

```powershell
auditpol /set /subcategory:"Kerberos Authentication Service" /success:enable /failure:enable
auditpol /set /subcategory:"Kerberos Service Ticket Operations" /success:enable /failure:enable
auditpol /set /subcategory:"Credential Validation" /success:enable /failure:enable
```

Additional configuration included:

- Enabling **Audit: Force audit policy subcategory settings to override audit policy category settings**
- Confirming that user `JNeutron` requires Kerberos pre-authentication
- Restarting the Splunk Universal Forwarder
- Applying domain policies using `gpupdate /force`

## ⚠️ False Positive Considerations

The following scenarios may generate similar Kerberos authentication activity and should be validated during investigation:

- User password mistakes against Kerberos-enabled services
- Repeated authentication attempts during application or service testing
- Administrative validation of accounts in a lab or support context
- Automated tooling generating authentication retries

Analysts should review source IP, target account, volume, and timing before escalation.

## 📝 Summary

This detection demonstrates end-to-end visibility of a Kerberos password spray attempt in an Active Directory environment.

Kerbrute generated the authentication request from the Kali attacker host, the Domain Controller recorded the Kerberos activity, and Splunk captured and surfaced the relevant event data for analyst review. This provides a practical detection workflow for identifying Kerberos-based credential guessing activity.
